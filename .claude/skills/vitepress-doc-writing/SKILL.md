# VitePress 文档写作

为本仓库编写和编辑文档时，只追求一个目标：让下一位读者愿意继续读下去。

## 什么时候使用这个 skill

当任务涉及以下内容时，使用这个 skill：

- 编写或重写 `docs/` 下的 Markdown 页面
- 优化教程的可读性、结构和阅读节奏
- 统一 VitePress 文档风格
- 补充示例、警告块、代码组、目录或其他 Markdown 扩展
- 把密集、偏硬的技术说明改写成更适合初学者阅读的学习材料

## 开始前先读的文件

在动手前，先阅读下面两个项目内参考文件：

1. `docs/guide/documentation-writing-standard.md`
2. `docs/guide/markdown-enhancements.md`

第一个文件是写作标准，第二个文件是 Markdown 扩展语法手册。

## 默认工作流

1. 先判断页面类型：教程页、参考页、FAQ，还是规划页。
2. 先让首屏有价值：
   - 标题清晰
   - 一句话讲明白这页解决什么问题
   - 需要时补一个 `tip` 或 GitHub alert 说明“这页怎么读”
3. 对长页面补 `## 页面导航` 和 `[[toc]]`。
4. 重新组织内容节奏：
   - 先讲结论
   - 再给例子
   - 再解释原理
   - 再提示风险或易错点
   - 最后给下一步
5. 有意识地使用 Markdown 扩展：
   - `tip / info / warning / details`
   - GitHub alerts，用于更强的编辑提醒
   - `code-group`，用于平台或包管理器差异
   - 行高亮，用于强调关键代码
   - `focus / ++ / -- / warning / error` 代码标记，用于提升扫读效率
   - 对重复内容使用导入代码片段或 Markdown include
   - 在确实能增强扫读体验时，使用少量语义稳定的 Emoji 作为视觉路标
6. 如果有助于阅读推进，在结尾加一个简短小结、检查表或下一页过渡。

## 按页面类型处理

### 教程页

- 开头先告诉读者读完能做成什么。
- 先给最小可运行路径，再讲深层原理。
- 用 `details` 把“主线内容”和“可选深入内容”分开。
- 至少补一处高价值警告，提醒最容易踩坑的点。

### 参考页

- 优先服务“快速查阅”，不是讲故事。
- 多用分组列表、简洁表格和短示例。
- 不要把答案埋在大段正文里。

### FAQ 页

- 使用 `[[toc]]`。
- 每个问题尽量单独放进一个 `details`。
- 存在平台差异时，用 `code-group` 包住命令。

## 本仓库的写作规则

- 以中文为主。
- 语气保持温和、务实、能增强读者信心。
- 在确实有帮助时，加入 “Go vs Java” 的对照视角。
- 站内链接使用无扩展写法。
- 示例尽量真实、可运行。
- 不要为了“功能多”而乱加扩展；每一个扩展都必须降低读者理解成本。
- Emoji 可以使用，但必须克制、稳定、有语义，不要让它们盖过内容本身。

## 完成前的质量门槛

提交前检查：

- 页面是否有 `title` 和 `description`
- 开头是否解释了“这页为什么值得读”
- 密集内容是否被标题或容器拆开
- 代码示例是否标了正确语言
- 关键操作是否一眼能找到
- 重复内容是否被抽取，而不是手动复制
- 进行过实质性文档修改后，`cd docs; pnpm docs:build` 是否通过

## 默认优先动作

- 把长段风险提示改成 `warning` 容器
- 把可选深入内容改成 `details`
- 把平台差异命令改成 `code-group`
- 给长文补 `[[toc]]`
- 在页面顶部补一句“这页怎么读”
- 在高层标题或关键提示处，补少量能增强扫读感的 Emoji

## 反模式

- 巨型文本墙
- 只贴代码，不解释读者为什么要关心
- 使用像“实现原理分析”这种抽象标题，却不给结果信号
- 把主流程和支线内容混在一起写
- 同一条说明在多个页面里手动重复
- 为了好看而堆格式，反而增加认知负担
- 把 Emoji 当成装饰贴纸乱用，导致页面显得轻浮或杂乱
