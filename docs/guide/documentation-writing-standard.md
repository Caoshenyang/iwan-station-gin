# 文档编写规范

![Iwan Station Logo](/images/logo.svg)

::: tip 这份规范的目标
这不是“排版规定”，而是一套帮助读者继续读下去的写作设计。好文档要做到三件事：先抓住眼球，再降低理解成本，最后让人愿意动手。
:::

## 北极星原则

| 原则 | 要做到什么 | 常见反例 |
|------|------------|----------|
| **先给答案** | 开头 30 秒内让读者知道这页解决什么问题 | 一上来先讲一大段背景史 |
| **一屏一重点** | 每个小节只解决一个核心问题 | 一个小节里同时讲概念、命令、排错、部署 |
| **让读者少猜** | 明确指出前置条件、输入输出和下一步 | 默认读者“应该懂”上下文 |
| **让页面有节奏** | 用 `TOC`、折叠块、代码组、警报块切分阅读负担 | 整页连续文本或巨型代码墙 |
| **先可用，再完整** | 优先给最小可运行路径，再补原理和细节 | 把最重要的操作埋在后面 |

> [!IMPORTANT]
> 教程页不是论文，也不是 API dump。读者愿不愿意继续学，往往取决于前 20 秒有没有看到“我现在就能用”的信号。

## Markdown 扩展怎么用

### 1. `frontmatter`

每一页都必须至少有：

- `title`
- `description`

建议：

- 标题直接写读者要搜索的关键词
- 描述用一句话交代“这页解决什么问题”

### 2. 标题锚点与 `[[toc]]`

- 长文默认加 `[[toc]]`
- 二级标题负责导航，三级标题负责拆解动作
- 不要出现“一个二级标题下面塞 800 行内容”

::: details 什么时候一定要加目录
- 页面内容超过 6 屏
- 同时存在多个步骤或多个子主题
- 这页未来会被频繁回查，而不是只读一次
:::

### 3. GitHub 风格警报 + 自定义容器

两类都可以用，但要有分工：

> [!TIP]
> 用于“建议怎么读”“推荐做法”“记忆口诀”。

> [!NOTE]
> 用于“补充背景”“和上一章/下一章的关系”。

> [!WARNING]
> 用于“高频踩坑”“生产环境风险”“路径和配置差异”。

::: info 使用建议
如果你想要中文标题和更统一的视觉语气，优先用 VitePress 的 `tip / info / warning / details` 容器；如果你想做更醒目的“编辑部提醒”，可以用 GitHub Alerts。
:::

### 4. 表格

表格只适合做“并列比较”，不适合塞长段解释。

推荐拿来写：

- 方案对比
- 参数说明
- 错误与解决方式
- Java vs Go 对照

### 5. 代码块

代码块必须满足三个标准：

1. 语言标记正确
2. 默认给最小可运行示例
3. 需要时标出重点行，而不是让读者自己找

#### 行高亮

```go{2,5-7}
func NewServer() *http.Server {
    engine := gin.Default()
    return &http.Server{
        Addr:    ":8080",
        Handler: engine,
        ReadTimeout:  30 * time.Second,
        WriteTimeout: 30 * time.Second,
    }
}
```

解释规则：

- `focus` 用来告诉读者“这一行最值得看”
- `++ / --` 用来表达改动前后差异
- `warning / error` 用来提醒风险和错误用法

#### 代码组

多平台、多包管理器、多调用方式时必须用 `code-group`，不要把不同命令硬堆在一起。

::: code-group

```bash [pnpm]
pnpm install
pnpm docs:dev
```

```bash [npm]
npm install
npm run docs:dev
```

```bash [yarn]
yarn
yarn docs:dev
```

:::

### 6. 折叠块 `details`

`details` 不是为了“藏内容”，而是为了控制阅读节奏。

适合折叠的内容：

- 可选方案
- 深水区原理
- 平台差异
- 长排错清单
- 不影响主流程的补充说明

不适合折叠的内容：

- 本页结论
- 第一条可运行命令
- 核心配置

### 9. 数学方程

只有在它真的能帮读者更快理解时才使用，不要为了高级而高级。

例如，这个教程站追求的理想阅读体验可以抽象成：

$$
Readability = \frac{Clarity \times Momentum \times Confidence}{Cognitive\ Load}
$$

你可以把它翻译成人话：

- 讲得越清楚
- 阅读节奏越顺
- 读者越有把握继续做下去
- 心智负担越低

页面就越“好读”。

### 10. 图片懒加载

架构图、流程图、终端截图都可以放，但要遵守三条：

- 图片必须服务理解，不是装饰噪音
- 图下方最好有一句“你应该看哪里”
- 同一页不要连续堆太多大图

## 怎么把页面写得更吸睛

### 开头先给一个“继续读下去”的理由

推荐开头结构：

1. 这页解决什么痛点
2. 读完你能做成什么
3. 建议怎么读

### 让文字有镜头感

可以多用这些写法：

- “先别急着记住全部细节，先抓住这一个判断标准”
- “如果你只记一件事，就记这一条”
- “这一步看起来小，但它会决定后面 3 章顺不顺”
- “第一次读先照着跑，第二遍再回来看原理”

### 让 Emoji 成为“视觉路标”，不是噪音

Emoji 可以让页面更有温度、更容易扫读，但前提是它承担了明确职责。

推荐使用场景：

- 放在高层标题里，帮助读者建立页面节奏
- 放在 `tip / warning / info` 标题附近，强化情绪信号
- 放在“学习目标”“常见问题”“快速开始”“发布检查”这类天然适合扫读的模块中

不推荐使用场景：

- 每个标题都带 Emoji，导致页面像海报墙
- 正文一两句就插一个表情，打断阅读
- 同一页使用太多不同风格的 Emoji，没有统一语义

建议控制：

- 普通页面每屏 `1-3` 个即可
- 长文优先把 Emoji 放在一级、二级标题或列表前，不要灌进整段正文
- 同一语义尽量固定同一 Emoji，形成读者记忆

#### 推荐语义映射

| Emoji | 适合表达什么 | 推荐场景 |
|------|--------------|----------|
| `🧭` | 导航、路线、章节入口 | 课程大纲、页面导航、阅读建议 |
| `🎯` | 目标、收益、结果 | 学习目标、先看结论 |
| `🚀` | 开始动手、快速启动 | 快速开始、第一步操作 |
| `🛠️` | 实操、配置、命令 | 环境要求、安装步骤、排错操作 |
| `💡` | 技巧、提醒、理解要点 | 阅读建议、关键理解 |
| `⚠️` | 风险、坑点、注意事项 | warning、易错点、生产提醒 |
| `✅` | 完成、推荐、正确做法 | 检查清单、最佳实践 |
| `📌` | 重点、结论、记忆点 | 小结、结论、本文重点 |

::: details 一个安全的默认原则
如果你不确定某个 Emoji 会不会显得太多，就问自己一句：去掉它之后，这一段会不会更难扫读？

- 如果答案是“不会”，那就删掉
- 如果答案是“会”，那它就有存在价值
:::

### 让页面有节拍，而不是一堵墙

推荐节拍：

- 结论
- 例子
- 解释
- 警告
- 回到主线

> [!TIP]
> 好教程不是“信息越多越厉害”，而是“读者越不容易掉队越厉害”。

## 禁忌清单

::: danger 千万别这样写
- 标题全是抽象名词，比如“实现原理分析”
- 一上来先贴 150 行代码，没有一句导读
- 多平台命令写在一个代码块里让读者自己猜
- 易错点埋在正文中部，没有任何提示块
- 同样的说明在 5 个页面里手动复制，后期全站一起过时
- 为了炫技乱用扩展，结果读者更分心
- 把 Emoji 当贴纸乱贴，结果页面比内容更抢戏
:::

## 发布前检查

::: details 文档发布前 10 项检查表

- 页面是否有清晰的 `title` 和 `description`
- 开头是否在 30 秒内告诉读者“这页值不值得读”
- 长文是否有 `[[toc]]`
- 是否至少有一处真正帮助读者避坑的提示块
- 多平台命令是否用了 `code-group`
- 关键代码是否做了行高亮或聚焦
- 可复用内容是否抽成了代码片段或 include
- 链接是否都能跳转
- 术语是否统一
- `pnpm docs:build` 是否通过

:::
